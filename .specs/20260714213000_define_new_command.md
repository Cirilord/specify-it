# Define New Command

## Objective

Define the first version of the `specify-it new` command so the project can deterministically scaffold new spec files from the repository configuration.

## Scope

- define the CLI contract for `specify-it new`
- define which inputs the command accepts in the first version
- define how the command derives the target file path and filename
- define the initial spec template generated by the command
- define how the command should behave when groups are enabled or disabled
- exclude LLM-authored content generation beyond the initial scaffold

## Design

The `new` command should be the deterministic entrypoint for creating a spec file that follows the repository contract before any LLM or human starts writing the actual content.

In the first version, the command interface should be:

- `specify-it new --title <title>`
- `specify-it new --title <title> --group <group>`

The command should accept the title through a required `--title` flag. The title should be the only semantic content passed through the CLI by default.

The first version should support the current repository root as the working directory and should load `specify-it.config.json` from that root before attempting to create a file. If the configuration file is missing or invalid, the command should fail with a clear error rather than guessing default behavior.

The command should load the repository configuration, resolve the configured spec root, naming strategy, format, language, and optional groups, then create a new spec file that matches those rules.

The command should not attempt to fully author the spec content. Its responsibility is to create the correct file in the correct location with the correct baseline structure.

The first version should create files only for repository configurations that use Markdown specs. If the repository is configured for another format such as `json`, `html`, or `xml`, the command should fail with a clear unsupported-format error until those scaffolds are explicitly designed. This keeps the command intentionally small while still aligning with the current configuration contract.

For Markdown output, the generated scaffold should include the standard section order already defined by the repository contract:

1. `# Spec Title`
2. `## Objective`
3. `## Scope`
4. `## Design`
5. `## Examples`
6. `## Acceptance Criteria`

The title provided through the CLI should be normalized for the filename slug and also rendered as a human-readable heading inside the generated file.

The filename should be derived from the configured naming strategy. For the first version, the required supported naming strategy should be `timestamp-slug`, because it matches the current project contract and existing spec examples. Other naming strategies may remain part of the config contract, but they do not need to be implemented by `new` until they are explicitly designed.

For `timestamp-slug`, the command should create filenames in the shape `YYYYMMDDHHMMSS_slug.md`. The timestamp should use the local system time at command execution. The slug should be lowercase, use hyphens between words, collapse repeated separators, and remove leading or trailing separators.

The `specs.groups` configuration should be optional. When it is omitted, the command should create the file directly under the configured specs root. When it is present, it should be defined as a simple list such as `["feat", "fix", "docs", "chore"]`, and the command should require `--group` with a value from that list.

If a target file already exists at the resolved path, the command should fail without overwriting the file. The first version should prefer safe behavior over automatic retries or suffix generation because spec filenames are meant to be stable and intentional.

The first version should keep the interface intentionally small. It should not accept the full body content through CLI arguments, because that would make shell usage fragile and would blur the boundary between deterministic scaffolding and LLM-assisted authoring.

The command output should report the file that was created so agents, scripts, and users can immediately open or modify it. On failure, the command should return a non-zero exit code and print a short message that explains whether the problem was missing config, invalid group input, unsupported format, unsupported naming, or an existing target file.

## Examples

Example CLI usage when groups are disabled:

- `specify-it new --title "bootstrap release workflow"`
- creates `.specs/20260714213000_bootstrap_release_workflow.md`

Example CLI usage when groups are enabled:

- `specify-it new --title "add config loader" --group feat`
- creates `.specs/feat/20260714213000_add_config_loader.md`

Example Markdown scaffold:

```md
# Bootstrap Release Workflow

## Objective

## Scope

## Design

## Examples

## Acceptance Criteria
```

## Acceptance Criteria

- `specify-it new` is defined as the command responsible for creating new spec files
- the first version uses the CLI shape `specify-it new --title <title>` with optional `--group <group>`
- the first version accepts a title through a required `--title` flag
- the command loads `specify-it.config.json` from the repository root and fails clearly when config is missing or invalid
- the command uses repository configuration to resolve root, format, naming, language, and optional groups
- the first version supports Markdown scaffolding and fails clearly for other configured formats
- the first version requires support for the `timestamp-slug` naming strategy
- the `timestamp-slug` filename shape is defined as `YYYYMMDDHHMMSS_slug.md`
- the slug normalization behavior is defined
- `specs.groups` is optional
- when groups are disabled, the file is created directly under the configured specs root
- when groups are enabled, the command requires `--group` and validates it against the configured list
- the command does not overwrite an existing target file
- the generated file includes the standard spec section order for the target format
- the command does not require or accept the full spec body as a normal CLI argument in the first version
- the command reports the created file path after scaffolding succeeds
- the failure modes and user-facing error categories are defined
